## 1. Концепция и роли B1/B2

### 1.1 Жизненный цикл скважины

```
[Приёмка скважины]
        |
[Установка датчиков]  → событие event_type='equip' в таблице events
        |
[ЭТАП НАБЛЮДЕНИЯ]     → period = obs_from .. obs_to
   - Записываем базовые данные давления (наши датчики)
   - Нет вбросов реагентов (или только тестовый)
   - Длительность: от суток до нескольких недель
        |
  (опционально)
[Тестовый вброс]      → событие event_type='reagent', но obs_to НЕ меняется
        |
[ЭТАП АДАПТАЦИИ]      → period = adapt_from .. adapt_to
   - Систематические вбросы реагентов
   - Нестабильный режим, подбор химии и периодичности
   - Может быть открытым (adapt_to = today)
        |
[ОПТИМИЗАЦИЯ]         → adapt_to .. бесконечно, статус скважины «Оптимизация»
   - Ежемесячный аналитический цикл (см. TZ_optimization_report.md)
   - Сравнение с B1, B2, оптимумом адаптации, предыдущим месяцем, историей
```

### 1.2 Базовые значения B1 и B2

| Метка | Полное название | Источник данных | Обязательность |
|-------|-----------------|-----------------|----------------|
| **B2** | Baseline наблюдения (наши данные) | `pressure_raw` → `pressure_hourly` + маски → `collect_stage_stats` | Рекомендован; создаётся кнопкой «Зафиксировать как baseline наблюдения» во вкладке настройки наблюдения |
| **B1** | Baseline заказчика | `well_daily` (выгрузка УзКорГаз) → `compute_period_stats` | Опционален; создаётся из вкладки «Исходные данные» кнопкой «Сохранить текущий период как baseline» |

**Отношение B1 к B2:** они независимы и взаимодополняют друг друга. B2 — то, что измерили мы сами; B1 — то, что говорит заказчик. Оба нужны для валидации: вкладка «Сравнение периодов» накладывает их поверх друг друга, чтобы доказать корреляцию.

**Хранилище:** таблица `customer_baseline` (поле `source`: `'observation'` для B2, `'customer'` для B1).

### 1.3 Правила работы с датами и тестовым вбросом

**Базовые ограничения:**
- `obs_from < obs_to` — блокирующая ошибка
- `adapt_from < adapt_to` — блокирующая ошибка
- `obs_to <= adapt_from` — нормальное состояние (gap разрешён)
- `adapt_from < obs_to` — перекрытие; система автоматически укорачивает `obs_to = adapt_from` с предупреждением, не блокирует

**Тестовый вброс (специальный сценарий):** пользователь сделал вброс в период наблюдения, но не считает это началом адаптации. Решение — двигать `obs_to` вручную: `obs_to` можно поставить позже первого события `event_type='reagent'`. Источник дат при этом должен быть `'manual'` (не `'events'`), иначе `suggest_stages_from_events` автоматически поставит `obs_to = reagent_first`. Предупреждение в UI: «obs_to ({дата}) позже первого вброса реагента ({дата}) — убедитесь, что это тестовый вброс, а не начало адаптации».

## 2. Исходные данные системы

| Таблица | Что содержит | Источник записей | Свежесть | Что делать если нет данных |
|---------|--------------|-------------------|-----------|---------------------------|

| `pressure_hourly` | Почасовые агрегаты (медиана по 60 мин) | `pressure_aggregate_service.py` | Задержка до 1 часа | Рассчитать из `pressure_raw` на лету через `_load_masked_hourly` |
| `pressure_masks` | Маски верифицированных/отфильтрованных диапазонов | Ручная разметка через `/pressure-masks` | Актуальны как справочник | Использовать данные без масок (предупреждение пользователю) |
| `well_daily` | Суточные показатели от заказчика (Q газ рабочий/суточный, P устье/шлейф, простои) | Ручная загрузка XLSX через парсер | Нерегулярно | Глава «Исходные данные» в PDF не формируется, вкладка 3 показывает «Нет данных заказчика» |
| `events` | События (вбросы, продувки, установка оборудования) | Telegram-бот + ручной ввод | Актуально | `suggest_stages_from_events` вернёт `obs_from=None`, источник дат переключится на `'manual'` |
| `well_status` | История этапов (Наблюдение, Адаптация, ...) | Ручной ввод администратора | Актуально | `validate_stages` вернёт `ok=False`, источник дат переключится на `'events'` или `'manual'` |
| `well_construction` | Технические данные скважины (choke_diam_mm, горизонт, перфорация) | Ручной ввод | Актуально | Без штуцера `collect_report_data` возвращает `ok=False` — **блокирующая ошибка**, отображается баннером |
| `customer_baseline` | Зафиксированные базовые периоды (B1, B2) | Кнопки сохранения на странице | Актуально | Сравнения с baseline не отображаются; страница работает в режиме «только наблюдение/адаптация» |
| `lora_sensors` | Реестр LoRa-датчиков со скважинами | Ручной ввод / `lora_sensors` роутер | Актуально | Блок «датчики» в вводном тексте §3 PDF будет пустым |
| `adaptation_report_state` | Сохранённое состояние формы по скважине | Автосохранение при изменениях | Актуально | Форма загружается с дефолтными значениями |

## 3. Левая панель «Настройка отчёта» — детальное ТЗ

### 3.1 Секция «Этапы скважины» (история)

**Назначение:** обзорная плитка хронологии этапов из `well_status`.

| Поле | Тип | Источник | Поведение |
|------|-----|----------|-----------|
| Плитки этапов | Read-only | `_load_all_stages(db, well_id)` → `well_status` | Цвет берётся из `STATUS_BY_LABEL[status]["color"]`. Открытый этап (dt_end=NULL) — пунктирная рамка + бейдж «активен» |
| Длительность | Число + «сут.» | Вычисляется: `(dt_end or now) - dt_start` | Показывать даже для открытых этапов |
| Примечание | Текст | `note` из `well_status` | Обрезать до 2 строк с ellipsis |

**Пустое состояние:** «Этапы в журнале статусов не найдены. Задайте даты вручную.»

**Definition of done:** все записи `well_status` для выбранной скважины отображаются в хронологическом порядке; открытый этап помечен визуально; нажатие на плитку подставляет даты этапа в поля наблюдения/адаптации.

---

### 3.2 Секция «Источник дат этапов»

**Три взаимоисключающих режима:**

| Режим | Откуда берутся даты | Когда применяется |
|-------|---------------------|-------------------|
| `status` | `validate_stages(db, well_id)` → `well_status` | Если в журнале есть записи Наблюдение + Адаптация |
| `events` | `suggest_stages_from_events(db, well_id)` → `events` | Если `status` не нашёл данные, но есть события установки оборудования и вбросов |
| `manual` | Поля ввода дат заполняются пользователем | Всегда доступен; активируется автоматически если оба авто-источника не дали результат |

**Чекбокс «Ограничить адаптацию N сут.»:** при включении поле `adapt_to` принудительно выставляется в `adapt_from + N дней`. Поле N: тип integer, диапазон 1–365, дефолт `DEFAULT_ADAPT_DURATION_DAYS = 10`.

**Поведение при переключении источника:**
1. Переключение → запрос `GET /api/adaptation-report/stage-data?well_id=...`.
2. Поля дат заполняются из выбранного источника.
3. Если выбранный источник не вернул данные → показать inline-предупреждение «Источник не найден, заполните даты вручную», остаться в режиме `manual`.

**Edge case — тестовый вброс:** при `source='events'` `obs_to` ставится на `reagent_first`. Если пользователь переключился в `manual` и вручную сдвинул `obs_to` позже `reagent_first` — показывать жёлтый баннер: «obs_to ({дата}) позже первого вброса реагента ({дата}). Если это тестовый вброс, убедитесь что adapt_from выставлен на следующий вброс.»

**Definition of done:** переключение режима немедленно обновляет поля дат (без нажатия «Пересчитать»); при переключении в `manual` поля редактируемы; при переходе обратно на `status`/`events` поля заменяются значениями источника с уведомлением.

### 3.3 Секция «Этап наблюдения»

| Поле | Тип | Дефолт | Валидация |
|------|-----|--------|-----------|
| `obs_from` | datetime-local | Из источника дат | Обязательное |
| `obs_to` | datetime-local | Из источника дат | `obs_to > obs_from`; `obs_to <= adapt_from` (предупреждение если нарушено, не блокировка) |
| Описание (textarea) | Строка | `well_status.note` для этапа Наблюдение | Необязательно |

**Кнопка «Зафиксировать как baseline наблюдения» (B2):**
- Активна только если пользователь admin.
- Вызывает `POST /api/adaptation-report/baselines/observation` с `{well_id, obs_from, obs_to, name="Этап наблюдения"}`.
- Сервис `save_observation_baseline` запускает `collect_stage_stats` за период наблюдения и сохраняет результат в `customer_baseline` с `source='observation'`.
- После успешного сохранения: блок `#obs-baseline-display` обновляется; показывается строка «B2 сохранён: Q мед. X тыс.м³/сут, ΔP мед. Y кгс/см², КИВ Z%» со ссылкой «удалить».
- **Текущая проблема (см. §7):** кнопка не реагирует на изменение дат — пользователь меняет `obs_from`/`obs_to`, нажимает кнопку, но в модальном окне не видно что именно будет сохранено. Требуется: перед POST показывать preview-строку «Будет сохранён период: {obs_from} – {obs_to}».

**Блок «Текущий B2»:** отображает последний сохранённый baseline с `source='observation'` для данной скважины. Если нет — серый текст «Baseline наблюдения не зафиксирован».

**Definition of done:** после нажатия кнопки B2 блок отображения немедленно обновляется без перезагрузки страницы; поля Q/ΔP/КИВ/даты отображаются корректно; повторное нажатие кнопки предлагает перезаписать или создать новый.

### 3.4 Секция «Этап адаптации»

| Поле | Тип | Дефолт | Валидация |
|------|-----|--------|-----------|
| `adapt_from` | datetime-local | Из источника дат | Обязательное; `adapt_from >= obs_from` |
| `adapt_to` | datetime-local | Из источника дат | `adapt_to > adapt_from`; допустим `adapt_to = today` для незакрытых этапов |
| Описание (textarea) | Строка | `well_status.note` для этапа Адаптация | Необязательно |

**Кнопка «Автоподбор дат»:**
- Вызывает `suggest_stages_from_events` повторно с текущим `adapt_days` параметром.
- Заполняет только поля адаптации (не трогает поля наблюдения).
- Если событий нет — предупреждение «Вбросы реагентов не найдены. Укажите даты вручную».
- **Текущая проблема (см. §7):** кнопка работает только если источник = `'events'`; при `source='manual'` ничего не происходит.

**Warning «Ручные даты ≠ Авто»:** показывается если текущие даты в полях отличаются от `stages_from_events.adapt_from`/`adapt_to`. Текст: «Ваши даты отличаются от автоподбора ({дата} – {дата}). Автоподбор основан на событиях вбросов реагентов.»

**Definition of done:** при изменении `adapt_from`/`adapt_to` вручную warning появляется если значение отличается от авто; автоподбор работает независимо от выбранного источника дат.

### 3.5 Секция «Оптимальный режим»

| Поле | Тип | Дефолт | Описание |
|------|-----|--------|----------|
| Режим | select: auto / manual / off | `auto` | `auto` — `detect_optimal_windows` выбирает лучшее окно; `manual` — даты задаются вручную; `off` — раздел оптимума отключён |
| Ширина окна (дни) | integer 1–30 | 3 | Только для режима `auto`; передаётся как `optimal_window_days` |
| `optimal_from` | datetime-local | Видим только при `manual` | |
| `optimal_to` | datetime-local | Видим только при `manual` | `optimal_to > optimal_from`; период должен лежать внутри `adapt_from..adapt_to` |

**Поведение при `auto`:** после «Пересчитать» список `optimal_candidates` (top-3) отображается во вкладке «Сравнение этапов». Первый кандидат автоматически выбирается как `optimal_regime`.

**Поведение при `manual`:** поля `optimal_from`/`optimal_to` обязательны; `detect_optimal_windows` не вызывается. Если вручную заданный период выходит за `adapt_from..adapt_to` — жёлтый баннер предупреждения.

**Definition of done:** переключение режима скрывает/показывает релевантные поля; при `off` раздел «Оптимальный режим» не появляется в вкладке «Сравнение этапов» и в PDF.

### 3.7 Кнопки «Пересчитать» / «Сформировать PDF»

**«Пересчитать» (`btn-recompute`):**
- Собирает все параметры формы: `{well_id, obs_from, obs_to, adapt_from, adapt_to, obs_description, adapt_description, optimal_mode, optimal_window_days, optimal_from, optimal_to, include_customer_chapter, customer_periods, sections}`.
- Вызывает `POST /api/adaptation-report/compute` с `with_charts=false` (быстрый режим, без PNG).
- Обновляет все вкладки: статистику, помесячную таблицу, список кандидатов оптимума.
- **Не** перегенерирует PDF.

**«Сформировать PDF» (`btn-preview`):**
- Вызывает `POST /api/adaptation-report/preview-pdf` с теми же параметрами + `with_charts=true`.
- Результат отображается в `<iframe id="ar-iframe">`.
- Блокирует кнопку на время генерации; spinner над iframe; таймаут предупреждения: 60 секунд.

**Definition of done:** «Пересчитать» занимает менее 10 секунд для периодов до 90 дней (без рендеринга PNG); «Сформировать PDF» показывает spinner и не блокирует остальной UI.

## 4. Правая панель — 5 вкладок

### 4.1 Вкладка «Сравнение этапов»

Источник данных: `collect_report_data(...)` через `POST /api/adaptation-report/compute`.

#### Блок A — Графики этапов

| Тип графика | PNG-файл | Инструмент | Что отображает |
|-------------|----------|------------|----------------|
| P трубное / P линейное | `adapt_p_{number}_obs.png` | `_render_stage_pressure_chart` | Почасовые медианы p_tube + p_line с маркерами событий |
| ΔP с трендом | `adapt_dp_{number}_obs.png` | `_render_stage_dp_chart` | ΔP = p_tube − p_line + линия тренда Theil-Sen + 95% CI |
| Q дебит | `adapt_q_{number}_obs.png` | `_render_stage_flow_chart` | Мгновенный дебит из `_compute_hourly_flow` |
| Комбинированный | `adapt_combined_{number}_obs.png` | `_render_stage_combined_chart` | P + ΔP + Q в 3-х строках на полную страницу |

Те же 4 графика генерируются для адаптации (тег `adapt`) и для оптимального режима (тег `optimal`).

**Пустое состояние:** если `pressure_hourly` за период пуст — блок графика заменяется плашкой «Нет данных давления за период {obs_from}–{obs_to}».

#### Блок B — Таблица сравнения Н / А / Опт

| Метрика | Источник | Единица |
|---------|----------|---------|
| Q медиана | `flow_median` из `collect_stage_stats` | тыс.м³/сут |
| Q среднее | `flow_avg` | тыс.м³/сут |
| ΔQ мед. (А − Н) | `comparison.delta_flow_median` из `build_comparison` | тыс.м³/сут + % |
| ΔP медиана | `dp_median` | кгс/см² |
| ΔΔP (А − Н) | `comparison.delta_dp` | кгс/см² + % |
| P труб. медиана | `p_tube_median` | кгс/см² |
| P лин. медиана | `p_line_median` | кгс/см² |
| КИВ | `utilization_pct` | % |
| Простой | `downtime_hours` | ч |
| Продувки всего | `purge_count` | шт |
| Вбросы реагента | `reagent_count` | шт |
| Накопленная добыча | `flow_cumulative` | тыс.м³ |

Колонка «Опт.» отображается только если `optimal_mode != 'off'` и `optimal_regime != None`.

Направление изменения (↑/↓/→): показывается рядом с ΔQ и ΔΔP. Зелёный ↑ если ΔQ > 0 или ΔΔP > 0; красный ↓ если отрицательный; серый → если |%| < 5.

**Пустое состояние:** если `ok=false` из compute → вместо таблицы красный баннер с `data.error`.

#### Блок C — Топ-3 кандидата оптимального окна

| Колонка | Источник |
|---------|----------|
| Период | `candidate.start` – `candidate.end` |
| Q медиана | `candidate.q_median` |
| Q mean | `candidate.q_mean` |
| CV (стабильность) | `candidate.cv_q` (чем меньше, тем лучше) |
| КИВ | `candidate.utilization_pct` |
| Продувок/сут | `candidate.purge_freq_per_day` |

Видим только если `optimal_mode != 'off'`. Клик на строку — заполнить поля `optimal_from`/`optimal_to` в левой панели и перейти в режим `manual`.

## 7. Что сейчас сломано или нелогично

### 7.1 Баг: `Well.choke_mm` в `/api/adaptation-report/our-segment`

**Файл:** `backend/routers/adaptation_report.py`, строка 601.

У модели `Well` нет поля `choke_mm` — оно хранится в `well_construction.choke_diam_mm`. `getattr` вернёт `None`, итого `choke_mm = None`. Endpoint `/our-segment` будет возвращать Q=None для всех точек.

**Исправление:** читать штуцер через `get_choke_mm(well_id)`.

### 7.2 Три пути расчёта Q — неконсистентность

В системе сосуществуют `_compute_hourly_flow`, `_compute_month_flow_from_raw`, `calculate_flow_rate`. `collect_stage_stats` считает агрегатные `flow_median`/`flow_avg` через `_compute_month_flow_from_raw`, а `chart_data[].q` через `_compute_hourly_flow`. Из-за этого медиана Q в таблице сравнения и медиана на графике могут расходиться.

**Требование:** для Chart.js-графиков — `_compute_hourly_flow`; для агрегатных метрик (median/avg/cumulative) — `_compute_month_flow_from_raw`. Не путать в подписях.

### 7.3 Дублирующий `comparison_service.py`

Сервис существует, во вкладке 4 используется через iframe. При переходе на нативный блок — подключить через `/api/adaptation-report/our-segment` + `compute_period_stats`.

### 7.4 Legacy `_linear_trend` импортируется но не используется

`backend/services/adaptation_report_service.py:31`. Убрать из импорта.

### 7.5 iframe вкладок не знает контекст формы

При смене скважины iframe не перезагружается атомарно. Даты из формы не передаются. **Минимальный fix:** при смене `currentWellId` все iframe с `data-empty` сбрасываются и перезагружаются с новым `?well=`.

### 7.6 Кнопка «Зафиксировать B2» — неясное состояние

После сохранения не показывает что именно зафиксировано. Требуется отображать период + Q мед / ΔP мед / КИВ сразу.

### 7.7 `include_customer_chapter or True` — всегда собирается глава

`backend/services/adaptation_report_service.py:2583`. Убрать `or True`. При «Пересчитать» (без PDF) передавать `include_customer_chapter=False`.

### 7.8 Автоподбор дат не работает при `source='manual'`

Кнопка должна работать независимо от выбранного источника. Текст подсказки: «Обновлены только поля адаптации. Источник дат остаётся ручной».

### 7.9 Нет graceful degradation при пустой `well_daily`

Вкладка 3 показывает пустую таблицу без сообщения. Требуется блок: «Данные заказчика не загружены. Используйте блок парсера для импорта XLSX.»

### 7.10 Помесячная статистика и история штуцера

`compute_monthly_stats` берёт последний штуцер из `well_construction`. Если штуцер менялся — Q за старые периоды неверен. Требование: предупреждение «Штуцер менялся: {история}. Q за ранние месяцы рассчитан с текущим штуцером — погрешность возможна».

## 8. Глоссарий

| Термин | Расшифровка |
|--------|-------------|
| **B1** | Baseline 1 — базовые значения по данным заказчика (из `well_daily`, источник `'customer'`) |
| **B2** | Baseline 2 — базовые значения по нашим датчикам за этап наблюдения (из `pressure_raw`, источник `'observation'`) |
| **ИРВ** | Интервал Реагентного Воздействия — период от вброса до следующего вброса/продувки |
| **Score** | Интегральный балл эффективности реагента в ИРВ (0–100), 7 метрик с весами |
| **КИВ** | Коэффициент Использования Времени = `working_hours / hours_with_data * 100` |
| **Q** | Дебит газа, тыс. м³/сут |
| **ΔP** | Перепад давления = `p_tube − p_line` (кгс/см²) |
| **Оптимальный режим** | Подпериод внутри адаптации с максимальным стабильным дебитом |
| **Theil-Sen** | Робастный метод оценки тренда (медианный slope), `_robust_trend` |
| **Mann-Kendall** | Тест на монотонность тренда. p-value < 0.05 — значимый |
