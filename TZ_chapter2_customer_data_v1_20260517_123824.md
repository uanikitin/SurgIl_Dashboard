# ТЗ: Глава 2 «Анализ исходных данных» в PDF отчёта адаптации

**Версия:** 2.0 (финальная)
**Дата:** 2026-05-17
**Источник UI:** страница `/customer-daily` (создание блоков) + wizard `/adaptation-report/wizard` (финальная сборка PDF)
**Целевой выход:** глава 2 PDF, шаблон `backend/templates/latex/adaptation_report.tex`

---

## 0. Терминология

**snapshot** = поле `data_snapshot` (JSONB) в таблице `customer_report_block`.

Это **JSON со структурированными данными** (массивы чисел, метрики,
тексты), которые были рассчитаны/показаны на странице в момент сохранения
блока. **Не PNG и не картинка.** Из этих данных:

- на странице рисуется Plotly-график;
- в PDF из тех же данных рисуется matplotlib-график (генерация на лету).

**Принцип:** snapshot — единственный источник правды для PDF. Backend
**не пересчитывает** данные из `well_daily` при сборке отчёта (за
исключением fallback'а для старых блоков без snapshot).

## 3. Принципы (главные правила)

1. **snapshot = единственный источник правды для PDF.** Backend **НЕ пересчитывает** данные из `well_daily` при сборке отчёта. Всё что показывалось пользователю на странице — сохраняется в `data_snapshot` и оттуда же берётся в PDF.
2. **UI-элементы в отчёт НЕ идут.** Исключения: загрузка xlsx, таймлайн, плашки реквизитов, индикаторы покрытия, кнопки, модалки. В отчёт идёт **только аналитический контент**: графики, таблицы, текстовые блоки анализа.
3. **Per-block настройки видимости** хранятся в `params.parts.{ключ: bool}`. По умолчанию все элементы блока включены.
4. **Один shared-модуль для общего функционала.** Если инструмент используется на нескольких страницах (как «выбор участков для сравнения» — на /well, в wizard'е, и на /customer-daily) — он реализован одним JS-файлом в `backend/static/js/` и подключается везде. Никаких дубликатов кода.
5. **Используем существующие функции — не переписываем и не дублируем.** Перед написанием нового кода проверяем что нет уже работающей реализации (на бэке, фронте или в shared-модулях). Если есть — используем её, при необходимости расширяя параметрами.
6. **Каждый блок отчёта обрамляется пользовательским текстом.** В начале блока — поле «вступительный комментарий», в конце — «заключительный комментарий». Оба хранятся в `params.prefix_note` / `params.suffix_note`. По умолчанию пусто; если пусто — в PDF не печатается. UI этих полей — компактные textarea в карточке блока, с автосохранением (debounce 600мс).

## 4. Состав `data_snapshot` блока `period_analysis` / `baseline`

(Структура `_v: 2`, формируется фронтом в `_buildPeriodSnapshot()`.)

```text
{
  // ─── Период и сводные метрики ───────────────────────────
  date_from, date_to, days, date_min, date_max,
  q_total_avg/median, q_working_avg/median,
  p_wellhead_median, p_flowline_median, dp_avg/median,
  shutdown_min_total, shutdown_days_count, q_trend,

  // ─── Данные для 4 графиков (массивы значений по суткам) ─
  chart: { dates, p_wellhead, p_annular, p_flowline, p_static, dp,
           q_gas_total, q_gas_working, shutdown_min },

  // ─── Таблицы ────────────────────────────────────────────
  describe:     [...],   // описательная статистика по параметрам
  monthly:      [...],   // помесячная агрегация + тренды
  monthly_desc: [...],   // помесячный анализ текстом

  // ─── Overlay UniTool (если был наложен на странице) ─────
  unitool: {
    first_date, last_date, days, equip_dt, dropped_pre_equip,
    choke_mm, has_flow, well_number,
    describe_map, by_month,
    daily: { dates, p_tube, p_line, dp, q }
  },

  // ─── Строгое сравнение UzKorGaz ∩ UniTool ───────────────
  strict_compare: {
    dates_from, dates_to, days,
    cust_q_total/q_working/dp/p_tube/p_line: { n, mean, median, min, max },
    our_q/dp/p_tube/p_line: { n, mean, median, min, max }
  },

  overlap_html   // готовый HTML 3 описательных блоков (для UI-модалки превью)
}
```

**Дополнительно в `params`** (хранятся отдельно, не в snapshot):

```text
{
  date_from, date_to, well_number,
  parts: { intro: true, description: true, ... },    // per-block тогглы
  prefix_note: "...",                                // пользовательский текст до
  suffix_note: "..."                                 // пользовательский текст после
}
```

## 5. Структура главы 2 в PDF

Глава = N подразделов, по одному на каждый блок с `in_report=true`.
Порядок: `baseline → period_analysis → criteria_rose → comparison`.

### 5.1. Подраздел `baseline` / `period_analysis`

Заголовок: «2.N. Период YYYY-MM-DD … YYYY-MM-DD».

Управляется per-block тогглами `params.parts.X` (порядок строго сверху
вниз — это же порядок в PDF):

| `parts.X` | Что в PDF | Источник | Статус |
|---|---|---|---|
| `prefix_note` | Пользовательский текст перед блоком | `params.prefix_note` | ✅ |
| `intro` | Сводка периода (1–2 предложения) | вычисляется из q/p/dp/shutdown полей snapshot | ✅ |
| `description` | Комментарий пользователя | `block.comment` | ✅ |
| `pressures_chart` | PNG: 4 давления + overlay UniTool | `snapshot.chart` + `snapshot.unitool.daily` | ✅ |
| `dp_chart` | PNG: ΔP + overlay UniTool ΔP | `snapshot.chart.dp` + `snapshot.unitool.daily.dp` | ✅ |
| `flow_dt_chart` | PNG: Q общий/рабочий + бары простоя + overlay UniTool Q | `snapshot.chart.{q_*,shutdown_min}` + `snapshot.unitool.daily.q` | ✅ |
| `monthly_chart` | PNG: помесячный групп. бар (Q ср./мед.) | `snapshot.monthly` | ✅ |
| `describe_table` | Таблица описательной статистики | `snapshot.describe` | ✅ |
| `monthly_table` | Таблица помесячных трендов | `snapshot.monthly` | ✅ |
| `monthly_descriptions` | Помесячный анализ текстом | `snapshot.monthly_desc` | ✅ |
| `overlap_text` | 3 текстовых блока: «Анализ UzKorGaz» / «Анализ UniTool» / «Совместный период» | строится `build_overlap_blocks(snapshot)` | ✅ |
| `strict_compare` | Таблица «Строгое сравнение UzKorGaz vs UniTool на пересечении дат» | `snapshot.strict_compare` | ✅ |
| `suffix_note` | Пользовательский текст после блока | `params.suffix_note` | ✅ |

### 5.2. Подраздел `criteria_rose`

Заголовок: «2.N. {title}».

Управляется тогглами `parts.{prefix_note, rose_chart, contributions_chart,
metrics_table, warnings, description, suffix_note}`. ✅ работает.

### 5.3. Подраздел `comparison`

Заголовок: «2.N. {title}».

Управляется тогглами `parts.{prefix_note, segments_table, chart,
description, suffix_note}`. ✅ работает.

### 5.4. UX сборки главы — live-превью + PDF (НЕ РЕАЛИЗОВАНО)

**Sticky-панель справа (40% ширины)** на странице `/customer-daily`.
Отображается когда есть хоть один блок с `in_report=true`.

**Два режима через табы в шапке панели:**

- **`📑 HTML`** — JS-рендер главы из snapshot блоков. Мгновенный, всегда живой при любом изменении (toggle `in_report`, toggle `params.parts.X`, изменение `prefix_note`/`suffix_note`/`comment`, изменение `title`, drag-reorder). Стилизован под PDF (шрифт серифный, отступы).
- **`📄 PDF`** — iframe с готовым PDF (xelatex рендер). Перерисовывается по кнопке `🔁 Пересобрать PDF`. Когда снапшот блоков изменился с последней сборки — показывается бэдж **«PDF устарел»** красным.

**Дополнительные элементы панели:**

- Кнопка `↗ В окно` — выносит превью в pop-out для работы на двух мониторах. Синхронизируется с основной страницей через `BroadcastChannel`.
- Кнопка `⬇ Скачать PDF` — доступна когда PDF собран.
- Бэдж сверху: `«в отчёт: X из Y блоков»`.

**Поведение на экранах <1024px:**

Sticky-панель сворачивается в всплывающую кнопку `📑 Превью главы` внизу справа. По клику разворачивается на весь экран. Закрывается ✕.

**Триггеры обновления:**

| Действие | HTML | PDF |
|---|---|---|
| Toggle `in_report` блока | мгновенно | бэдж «устарел» |
| Toggle `params.parts.X` | мгновенно | бэдж «устарел» |
| Редактирование `prefix_note`/`suffix_note`/`comment` | через 500мс debounce | бэдж «устарел» |
| Drag-reorder блоков | мгновенно | бэдж «устарел» |
| Создание/удаление блока | мгновенно | бэдж «устарел» |
| Клик `🔁 Пересобрать PDF` | — | пересборка (5-12 сек) |
| Клик `↗ В окно` | синхронизируется | синхронизируется |

## 11. План реализации Two-Panel (УТВЕРЖДЁН 2026-05-17)

**Принцип:** делать строго по списку, по порядку, с проверкой после каждого этапа.
**Откат:** перед началом — git checkpoint, чтобы любой этап можно было откатить.

### Этап 1 — Подготовка
- Git commit текущего состояния (точка возврата).
- Снять screenshot текущей страницы для сравнения «было/стало».

### Этап 2 — Вынос загрузки XLSX на отдельную страницу
- Создать route `GET /customer-daily/upload` + шаблон `customer_upload.html`.
- Перенести туда: drag-and-drop, диалог дубликатов «Перезаписать/Только новые/Отмена», POST `/upload-check` + `/upload`, сводку результата.
- После успешной загрузки — auto-redirect на `/customer-daily?well=...`.
- На главной `/customer-daily` — оставить только плитку-ссылку `[📁 Загрузить XLSX ↗]` в верхней полосе.

### Этап 3 — Верхняя sticky-полоса
- Sticky сверху: `[Скв. №▼] [📌 B1: дата-диапазон или «нет»] [📁 Загрузить XLSX ↗] [хранилище: 4200 строк · 18 скв.]`.
- Высота ~50px. Всегда видна при скролле.

### Этап 4 — Удалить старые секции
Удалить с главной страницы (вынесено наверх или вообще не нужно):
- Секция «1. Загрузить XLSX» (→ отдельная страница).
- Секция «Хранилище данных заказчика» (→ плашка в шапке).
- Секция «2. Выбор скважины» (→ select в шапке).
- Секция «📌 Утверждённый B1» (→ бейдж в шапке).
- Секция «3. Базовый период (до начала работ)» (→ кнопка «✓ Утвердить как B1» под результатом «Периода анализа»).
- Кнопка «Перейти к отчёту об адаптации →» (удаляется — навигация в шапке wizard'а).

### Этап 5 — Two-Panel layout 60/40
- CSS grid: главный контейнер делится на `.cd-left` (60%) и `.cd-right` (40%).
- Правая часть `.cd-right` — `position: sticky; top: 60px;` (под верхней полосой).

### Этап 6 — Левая панель (60%)
- **Секция «Период анализа»** — всегда развёрнута. Содержит: таймлайн `cdtl`, поля С/по, чекбокс UniTool, кнопку «▶ Показать аналитику», результат (4 графика + 2 таблицы + помесячный текст), кнопки `[☑ Добавить в отчёт]` + `[✓ Утвердить как B1]`.
- **Аккордеон «📊 Диагностика по критериям»** (роза) — `<details>`, **свёрнут по умолчанию**, без сохранения состояния.
- **Аккордеон «🔁 Сравнение участков»** (drag-select cmp_segments.js) — `<details>`, **свёрнут по умолчанию**, без сохранения состояния.
- **Секция «Прикреплено к отчёту»** — список карточек блоков (как сейчас, с галочками `in_report`, тогглами `parts`, prefix/suffix_note).

### Этап 7 — Правая sticky-панель (40%)
Структура сверху вниз:
- **Шапка панели:** бэйдж `[в отчёт: X из Y блоков]`.
- **Табы:** `[📑 HTML] [📄 PDF]` — переключение режима превью.
- **HTML-таб:** мгновенный JS-рендер главы из snapshot блоков (использует существующую `renderChapterPreview()`).
- **PDF-таб:** iframe с PDF + бэдж `«PDF устарел»` (красный когда блоки менялись с последней сборки, зелёный когда актуален).
- **Кнопки внизу:** `[🔁 Пересобрать PDF]` `[⬇ Скачать]` `[↗ В окно]`.

### Этап 8 — Логика «PDF устарел»
- Любое изменение блока (toggle `in_report`, toggle `parts`, правка `prefix/suffix_note`, `comment`, `title`, drag-reorder, создание/удаление) → бэдж становится красным.
- Клик `[🔁 Пересобрать PDF]` → POST `/api/adaptation-report/preview-pdf` с `only_chapter='customer_data'` → через 5-12 сек новый PDF в iframe → бэдж зелёный.

### Этап 9 — Pop-out окно «↗ В окно»
- Новый route `GET /customer-daily/preview-popout?well=N`.
- Шаблон — только sticky-панель на весь экран (без левой части).
- Синхронизация с основной страницей через `BroadcastChannel('cd-chapter-preview-{well}')`: при изменении блоков на основной странице → popout мгновенно перерисовывает HTML-превью.
- **Pop-out — ТОЛЬКО ДЛЯ ПРОСМОТРА**, без галочек и редактирования.

### Этап 10 — Responsive (<1024px)
- Two-Panel → одна колонка (`.cd-left` занимает 100%, `.cd-right` уходит).
- Sticky-панель сворачивается в всплывающую кнопку `[📑 Превью главы]` в правом нижнем углу.
- По клику — разворачивается на весь экран. Закрывается крестиком.
- На <768px (смартфон) — popup занимает весь viewport.

### Этап 11 — Улучшения (УТВЕРЖДЕНЫ ВСЕ)
1. **Drag-reorder блоков** (изменение `sort_order`) — оператор перетаскивает карточки в списке «Прикреплено к отчёту» → меняется порядок в PDF.
2. **Подсветка блока** в HTML-превью при наведении на карточку (визуальная связь карточка ↔ блок в превью).
3. **Кнопки «Свернуть всё / Развернуть всё»** для блоков `<details>⚙ Что включить в PDF</details>` — когда блоков много, чтобы не кликать каждую.
4. **Цветовая рамка** карточки блока — по типу: `baseline=жёлтый`, `period_analysis=зелёный`, `comparison=фиолетовый`, `criteria_rose=красный`.
5. **Inline-редактирование заголовка** карточки — клик по title → редактируется на месте (сейчас отдельный input).

### Этап 12 — Регрессионный тест
- Drag-select на детальных графиках cmp_segments — работает.
- Таймлайн `cdtl` — работает.
- Сохранение блоков всех 4 типов (baseline, period_analysis, criteria_rose, comparison) — работает.
- Wizard `/adaptation-report/wizard` шаг 2 (iframe) — показывает новый layout.
- PDF собирается со всеми элементами (overlay UniTool, 3 текстовых блока, strict_compare, prefix/suffix_note).
- На <1024px sticky сворачивается во всплывающую кнопку.

## 12. Соглашения по умолчанию

- **Состояние аккордеонов** (свёрнут/развёрнут) — НЕ запоминается. При каждой загрузке страницы: «Период анализа» развёрнут, «Роза» и «Сравнение» свёрнуты.
- **Pop-out окно** — только для просмотра. Все галочки/редактирование/создание блоков — только на основной странице.
- **Сохранение всех изменений** (галочки, текст полей, перетаскивание) — автоматическое, через debounce 500-600мс. Без явных кнопок «Сохранить».
- **Live-превью HTML** обновляется мгновенно. Live-превью PDF — только по кнопке (пересборка ~5-12 сек).

## 10. Итог: что реально сделано в коде

**Сделано (8 из 11 пунктов §6):**

1. **Backend полностью готов**: snapshot читается как источник правды; рендерер графиков принимает UniTool overlay; helper `build_overlap_blocks(snapshot)` собирает 3 структурированных блока; `_format_*_block` для всех 3 типов блоков прокидывает `parts` + `prefix_note` + `suffix_note`.
2. **LaTeX-шаблон полностью обновлён**: каждая секция в каждом из 3 типов блоков обёрнута в `\BLOCK{ if p.parts.X }`; добавлены секции prefix_note/suffix_note (curs); добавлены 3 описательных блока; добавлена таблица «Строгое сравнение».
3. **UI карточек блоков**: 2 textarea (prefix/suffix) с пунктирной рамкой; collapsible `<details>⚙ Что включить в PDF</details>` с расширенным списком тогглов; debounced автосохранение на изменение.

**НЕ сделано (3 пункта §6):**

- §6.8: sticky-панель live-превью с табами HTML/PDF и бэджем «PDF устарел».
- §6.9: pop-out окно превью через `BroadcastChannel`.
- §6.10: responsive media-queries (<1024px / <768px).
- §6.11: перепроектирование структуры страницы Two-Panel + вынос загрузки XLSX на `/customer-daily/upload` + аккордеоны для розы/cmp.

**Критическая зависимость для работы изменений в браузере:**

- Сервер uvicorn должен быть **перезапущен** (он держит старый Python-код в памяти).
- Браузер должен сделать **hard reload** (`Cmd+Shift+R` / `Ctrl+Shift+R`) — иначе закеширован старый HTML/JS.
