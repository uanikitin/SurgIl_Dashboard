# Техническое задание: Контур «Оптимизация» — ежемесячный отчёт по скважине

**Версия:** 1.0 | **Дата:** 07.05.2026 | **Проект:** SurgIl Dashboard

> **Связанные документы:** `TZ_adaptation_report.md` описывает страницу `/adaptation-report` и контур этапов Наблюдение/Адаптация. Настоящее ТЗ описывает 5-ю вкладку этой страницы и сопутствующий месячный документ.

---

## 1. Концепция

### 1.1 Место в жизненном цикле

```
[Адаптация закрыта]
        ↓
[Администратор переводит скважину в статус «Оптимизация»] (well_status)
        ↓
[ОПТИМИЗАЦИЯ] — постоянный режим работы скважины
        |
        ├── 1-го числа каждого месяца — авто-формирование draft месячного отчёта
        ├── Инженер проверяет, корректирует описания/рекомендации
        ├── Утверждает (generated) → подписывает (signed) → отправляет (sent)
        └── Архивирует (archived)
```

### 1.2 Цель контура

Дать инженеру по выбранной скважине ежемесячно:

1. **Зафиксировать** показатели работы за календарный месяц (Q, ΔP, КИВ, простои)
2. **Оценить эффективность реагентного воздействия** в этом месяце (статистика вбросов, расход химии, ИРВ-Score, лучшие/худшие интервалы)
3. **Сравнить** с пятью базами одновременно: B1, B2, оптимум адаптации, предыдущий месяц, среднее по всем предыдущим месяцам оптимизации
4. **Увидеть динамику** — куда движется скважина за всё время оптимизации
5. **Получить рекомендацию** — продолжать режим / скорректировать частоту / сменить ПАВ / назначить повторную адаптацию
6. **Выпустить PDF** для заказчика, прошедший цикл draft → generated → signed → sent

### 1.3 Что НЕ покрывает этот контур

- Анализ периода Наблюдения и Адаптации (см. `TZ_adaptation_report.md`)
- Создание B1 и B2 (создаются на этапе адаптации, здесь только читаются)
- Подбор оптимального режима в момент адаптации (`detect_optimal_windows`) — здесь используется уже найденный

## 2. Жизненный цикл месячного документа

### 2.1 Тип документа

Новый `DocumentType` (запись в БД, не код):

| Поле | Значение |
|------|----------|
| `code` | `optimization_monthly` |
| `name_ru` | Месячный отчёт об оптимизации |
| `requires_well` | true |
| `requires_period` | true (period_year + period_month, period_start = 1-е число, period_end = последнее) |

### 2.2 Статусы и переходы

Используем существующий контракт документов (`backend/documents/models.py:121`):

| Статус | Эмодзи | Что разрешено | Переход «Сформировать PDF» |
|--------|--------|---------------|----------------------------|
| `draft` | 📝 | Редактирование описаний и рекомендаций; пересчёт; удаление | **Перезаписывает** существующий PDF и LaTeX-источник |
| `generated` | 📄 | Просмотр, скачивание, подписание | Создаёт **новую версию** (`parent_id`, `version+1`) в статусе draft |
| `signed` | ✅ | Просмотр, отправка | Создаёт новую версию |
| `sent` | 📧 | Просмотр, архивация | Создаёт новую версию |
| `archived` | 📦 | Терминальный, только просмотр | — |
| `cancelled` | ❌ | Терминальный, только просмотр | — |

**Уникальность черновика:** на одну (well_id, period_year, period_month) допускается **не более одного** документа в статусе draft. Попытка создать второй draft при наличии существующего → перезапись (см. логику `auto_create.py:206`). После перехода в generated этот «слот» освобождается, новый draft создаётся как новая версия.

### 2.3 Версионирование

Используем существующие поля:
- `Document.version` — целое, инкрементируется на каждой новой версии
- `Document.parent_id` — FK на предыдущую версию
- `Document.children` — relationship для построения дерева версий

Версия 1 — оригинал (после первого `generated`). Версия 2 — после правок в подписанном или отправленном. UI показывает все версии в карточке месяца с пометкой текущей.

### 2.4 Подписи

Используем существующую таблицу `DocumentSignature`. Роли для месячного отчёта:

| `role` | `role_title_ru` | Кто подписывает |
|--------|-----------------|-----------------|
| `executor` | Представитель Исполнителя | Технолог СУРГИЛ |
| `client` | Представитель Заказчика | Представитель УзКорГаз |
| `geologist` | Геолог | Геолог скважины |

### 2.5 Уведомления

Используем `documents/services/notification_service.py` без изменений. Триггеры:

- Переход в `generated` → уведомление в Telegram «Готов отчёт за {месяц} по скв {номер}»
- Переход в `sent` → лог в `models_notifications` (как у Актов)

## 4. Состав карточки месяца — 6 разделов

### 4.1 Раздел A — Производительность

| Метрика | Источник | Единица |
|---------|----------|---------|
| Q среднее / медиана / мин / макс | `well_month_stats.q_*` | тыс.м³/сут |
| ΔP медиана | `well_month_stats.dp_median` | кгс/см² |
| P трубное / линейное медиана | `well_month_stats.p_tube_median`, `p_line_median` | кгс/см² |
| КИВ | `working_hours / hours_with_data * 100` | % |
| Простой | `downtime_hours` | ч |
| Накопленная добыча за месяц | `q_cumulative` | тыс.м³ |
| Тренд Q внутри месяца | `q_slope_per_day` + Mann-Kendall p-value | ↑/↓/→ |

Графики: Q(t) с маркерами событий, ΔP(t), P_tube/P_line(t), КИВ-распределение.

### 4.2 Раздел B — Реагентный блок

| Метрика | Источник |
|---------|----------|
| Количество вбросов | `events` где `event_type='reagent'` в окне месяца |
| Интервалы между вбросами | min / median / max / CV — вычисляется из `events.dt` |
| Расход химии всего | сумма `events.quantity` |
| Расход по типам ПАВ | группировка по `events.reagent` (литры/кг по справочнику `ReagentCatalog`) |
| ИРВ-Score | таблица всех ИРВ месяца + распределение Score (median, p25, p75) |
| Лучший ИРВ месяца | строка с max Score + параметры режима |
| Худший ИРВ месяца | строка с min Score + что отличало |

Источник ИРВ: `analyze_reagent_effectiveness(well_id, period_start, period_end)` → результаты кэшируются в `irv_cache` (см. §5).

ИРВ-таблица сохраняется как `DocumentItem` записи (одна строка = один ИРВ) с полями `event_id`, `reagent_name`, `event_time`, `quantity`, `notes` (Score и метрики). Это переиспользует существующую структуру.

### 4.3 Раздел C — Сравнения (5 колонок)

| Метрика | Текущий месяц | vs B2 | vs B1 | vs Опт. адаптации | vs Пред. месяц | vs Среднее истории |
|---------|---------------|-------|-------|-------------------|----------------|--------------------|
| Q медиана | … | Δ + % | Δ + % | Δ + % | Δ + % | Δ + % |
| ΔP медиана | … | … | … | … | … | … |
| КИВ | … | … | … | … | … | … |
| Score среднее ИРВ | … | — | — | … | Δ | Δ |

**Источники баз:**
- B1, B2 — `customer_baseline` через `list_baselines(well_id)`
- Оптимум адаптации — последний `customer_baseline` с типом `'adaptation_optimal'` (новый source-tag) или metadata в существующей записи; уточнить при реализации
- Пред. месяц — `well_month_stats` за `(year_month - 1)`
- Среднее истории — агрегат всех `well_month_stats` со статусом «оптимизация» до текущего месяца

**Цветовая разметка:** зелёный ↑ при положительном ΔQ или ΔКИВ; красный ↓ при отрицательном; серый → при |%| < 5.

**Архитектурный задел (UI отложен):** API сравнения принимает параметр `compare_against` с типами `'baseline_b1'`, `'baseline_b2'`, `'baseline_optimal'`, `'previous_month'`, `'history_average'`, `'arbitrary_period'`. Тип `arbitrary_period` принимает (date_from, date_to) — реализуется в API сразу, в UI добавляется позже как фитча «Сравнить с произвольным периодом».

### 4.4 Раздел D — Динамика (longitudinal)

Графики «значение по месяцам оптимизации» с маркером текущего месяца:

- Q медиана по всем месяцам + линия B2
- ΔP медиана по всем месяцам
- Частота вбросов (вбросов/сут)
- Расход химии (л/мес или кг/мес) с разбивкой по типам ПАВ (stacked bar)
- Среднее Score ИРВ по месяцам — ключевой индикатор «не выгорает ли химия»

Источник: агрегация всех `well_month_stats` для скважины, упорядоченных по `year_month`.

Авто-вывод по динамике (текстовая строка): «За последние 6 месяцев Q стабильно (CV={n}%)» / «За последние 3 месяца Q снижается на {n}/мес» / «Score реагента деградирует — 75 → 58 за полгода».

### 4.5 Раздел E — Эффективные периоды месяца

Аналог `detect_optimal_windows` в окне месяца — топ-3 окна с лучшим Q при минимальном CV. Для каждого окна:
- Период (дата от / дата до)
- Q медиана / среднее / CV
- КИВ
- Сколько вбросов произошло в окне
- Какие типы ПАВ применялись

Клик по окну — раскрывает график Q/ΔP с подсветкой именно этого диапазона.

### 4.6 Раздел F — Рекомендация и заключение

Авто-генерация по правилам (формула, веса и пороги фиксируются в коде сервиса; см. §10):

| Условие | Рекомендация |
|---------|--------------|
| Q vs B2 ≥ +10% и стабильно ≥ 3 мес | Стабильно — продолжать режим |
| Q vs B2 положительно, но снижается ≥ 5%/мес | Тревога — увеличить частоту вбросов |
| Score ИРВ снизился на ≥ 20% за 3 мес | Сменить ПАВ (привести 2–3 кандидата по реестру) |
| Q vs B2 < 0 в течение 2+ месяцев | Назначить повторную адаптацию |

Инженер может **переопределить** авто-рекомендацию вручную в режиме draft. Финальная версия сохраняется в `Document.notes` или `Document.meta.recommendation`.

## 5. Расчётный кэш и инвалидация

### 5.1 Таблица `well_month_stats`

Назначение: хранить агрегаты на месяц, чтобы сравнения и динамика не пересчитывались с нуля каждый раз.

| Поле | Тип | Описание |
|------|-----|----------|
| `well_id` | FK Well | |
| `year` | int | |
| `month` | int (1–12) | |
| `q_median, q_mean, q_min, q_max` | float | тыс.м³/сут |
| `dp_median, p_tube_median, p_line_median` | float | кгс/см² |
| `kiv_pct` | float | |
| `working_hours, downtime_hours, hours_with_data` | float | |
| `q_cumulative` | float | тыс.м³ |
| `q_slope_per_day, q_mk_pvalue` | float | тренд |
| `purge_count, reagent_count` | int | |
| `chem_consumption` | JSONB | `{ "ПАВ-1": {liters: 12.3, kg: null}, ... }` |
| `interval_stats` | JSONB | `{ min, median, max, cv }` секунд между вбросами |
| `score_median, score_count` | float, int | |
| `score_distribution` | JSONB | `{ p25, p50, p75, min, max }` |
| `best_window` | JSONB | `{ start, end, q_median, kiv }` |
| `computed_at` | timestamp | когда рассчитано |
| `source_fingerprint` | text | хэш входных данных (см. §5.3) |
| `needs_recompute` | bool | флаг устаревания |

Уникальный ключ: `(well_id, year, month)`.

### 5.2 Таблица `irv_cache`

Назначение: хранить ИРВ и их Score, чтобы:
- помесячная динамика Score не требовала пересчёта 24 месяцев каждый раз
- ИРВ-таблица в карточке месяца открывалась мгновенно

| Поле | Тип |
|------|-----|
| `well_id` | FK Well |
| `irv_start, irv_end` | timestamp |
| `reagent_name` | text |
| `score` | float (0–100) |
| `metrics` | JSONB (7 расширенных метрик) |
| `event_id` | FK на события вброса (опционально) |
| `computed_at` | timestamp |
| `source_fingerprint` | text |
| `needs_recompute` | bool |

Уникальный ключ: `(well_id, irv_start)`.

### 5.3 Source fingerprint

Хэш входных данных, используемый для определения «изменилось ли что-то с момента последнего расчёта». Состав хэша:
- max(`pressure_raw.ts`) в окне расчёта
- max(`pressure_masks.updated_at`) для масок, пересекающихся с окном
- max(`events.updated_at`) в окне для типов `reagent`, `purge`
- значение `well_construction.choke_diam_mm` на дату конца окна

При пересчёте: если текущий fingerprint равен сохранённому → данные актуальны, флаг `needs_recompute=false`.

### 5.4 Инвалидация (выставление `needs_recompute=true`)

| Триггер | Что флагается |
|---------|---------------|
| Новые/удалённые точки `pressure_raw` в окне месяца | `well_month_stats(this_month)` + ИРВ окна |
| Изменение масок, пересекающих окно | то же |
| Смена `well_construction.choke_diam_mm` | все месяцы со старым штуцером |
| Добавлено/удалено событие `reagent`/`purge` в окне | `well_month_stats(this_month)` + соседние ИРВ |
| Изменены глобальные веса/пороги Score | весь `irv_cache` (форс-инвалидация) |

Реализация триггеров: hook'и в существующих сервисах (`pressure_import_sqlite`, `pressure_filter_service`, событийные ручки `events`) выставляют флаг. Конкретные точки внедрения — в карте изменений после ревью ТЗ.

### 5.5 UI индикаторы актуальности

| Состояние | Индикатор на канбан-карточке |
|-----------|-------------------------------|
| `needs_recompute=true` | 🟡 «Требует пересчёта» с подсказкой что изменилось |
| `needs_recompute=false` и `computed_at` < 7 дней | 🟢 «Актуально» |
| Запись отсутствует (нет `well_month_stats` для месяца) | ⚪ «Не считалось» |

Кнопки:
- На карточке месяца: «Пересчитать»
- На вкладке «Оптимизация» сверху: «Пересчитать всё устаревшее по скважине»

В PDF в подвале строка: «Расчёт от {computed_at:DD.MM.YYYY HH:MM}». В подписанной версии эта дата заморожена.

## 6. Режимы работы: Авто, Ручной, Редактирование

### 6.1 Авто (cron)

Скрипт `scripts/run_optimization_monthly.py` (по аналогии с `scripts/run_pressure_update.py`):
- Запуск через launchd/cron 1-го числа каждого месяца в 03:00 локального времени
- Проходит по всем скважинам со статусом «Оптимизация»
- Для каждой:
  1. Считает `well_month_stats` за прошедший месяц
  2. Считает/обновляет `irv_cache` для ИРВ прошедшего месяца
  3. Создаёт `Document(type=optimization_monthly, status=draft)` если ещё нет
  4. Заполняет `DocumentItem` строками по ИРВ
  5. Генерирует LaTeX → PDF
  6. Шлёт уведомление в Telegram о готовности draft (опционально, конфигом)

### 6.2 Ручной

Кнопка «Сформировать отчёт за месяц» на вкладке «Оптимизация»:
- Выбор скважины (из контекста страницы) и месяца (выпадающий список доступных)
- Та же логика что и авто, но триггер от пользователя
- Можно для любого прошедшего месяца, не только последнего

### 6.3 Редактирование (статус draft)

Что можно править на странице карточки месяца:
- Текстовое описание месяца (нарратив)
- Рекомендация (переопределение авто-генерации)
- Чекбоксы разделов PDF (какие из 6 разделов включать)
- Подписи (роли и имена подписантов — фактическое подписание происходит на переходе в `signed`)

Что нельзя править никогда (только пересчитать):
- Сами числовые метрики `well_month_stats` — пересчитываются из исходных данных
- ИРВ Score — пересчитывается из `irv_cache`

После «Сформировать PDF» в draft → файл перезаписывается, документ остаётся в draft.

После «Утвердить» → статус `generated`, любые правки создают новую версию.

## 11. Что не так в текущем коде, что нужно учесть до Оптимизации

Эти проблемы блокируют корректность месячного отчёта; должны быть исправлены до или вместе с реализацией ТЗ-2:

| Источник | Проблема | Влияние на Оптимизацию |
|----------|----------|------------------------|
| ТЗ-1 §7.1 | `Well.choke_mm` отсутствует, `getattr` возвращает None | Q=None во всех месяцах при использовании `/our-segment` |
| ТЗ-1 §7.2 | Три пути расчёта Q расходятся | Медиана Q в таблице и на графике месяца разные |
| ТЗ-1 §7.10 | История штуцера не учитывается | Q за старые месяцы оптимизации некорректен при смене штуцера |
| ИРВ | `analyze_reagent_effectiveness` пересчитывает каждый раз | 24 месяца × ИРВ-расчёт = тяжёлая операция; обязательно кэшировать |
| Score веса | Сейчас зашиты в код, документированы в `reagent_effectiveness_service.py` | При изменении весов нужна форс-инвалидация всего `irv_cache` (зафиксировать в коде сервиса) |

## 12. Definition of done

1. Скважина в статусе «Оптимизация» имеет открытую вкладку 5 на `/adaptation-report` со списком месяцев и индикаторами актуальности
2. Авто-cron 1-го числа создаёт draft месячного отчёта по всем активным скважинам
3. Карточка месяца открывает все 6 разделов с реальными данными из кэша
4. Сравнение текущего месяца с 5 базами (B1/B2/Опт/Пред/История) отображается в таблице с цветовой разметкой
5. PDF месячного отчёта проходит цикл draft → generated → signed → sent → archived через канбан, с версионированием и подписями
6. Изменение исходных данных (новые точки давления, маски, события, штуцер) корректно выставляет `needs_recompute=true` для затронутых месяцев
7. ИРВ Score одного месяца открывается мгновенно из кэша; пересчёт ИРВ за 12 месяцев укладывается в таймаут (≤30 сек)
8. Документ типа `optimization_monthly` появляется в общем канбане `/documents/` наряду с Актами

## 13. Out-of-scope (для первой версии)

- Многоскважинный сравнительный месячный отчёт (один отчёт по группе скважин)
- Произвольный период сравнения (UI; в API оставлена возможность через `compare_against='arbitrary_period'`)
- Сравнение типов ПАВ между собой как самостоятельная аналитика (выйдет позже как отдельная страница)
- Прогнозирование (когда Q просядет ниже B2, когда Score деградирует до критического)
- Автоматическая отправка подписанного PDF заказчику без участия инженера
- Миграция существующих исторических данных в `well_month_stats` за прошлые годы (только текущий и вперёд)
