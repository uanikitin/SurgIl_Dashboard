# ТЗ: Плиточный паттерн глав отчёта (интервал → плитки-анализы → модалка → снапшот-плитки → drag-n-drop)

Статус: SPEC (на согласовании/реализации). Режим работы: IMPLEMENT после approve.
Эталонная глава: **Адаптация (шаг 5 визарда)**. Паттерн проектируется как переиспользуемый
для глав **Наблюдение** (шаг 3) и **Заказчик** (`/customer-daily`, PROTECTED — только после явного
разрешения).

Источник: AUDIT UIDesigner 2026-06-03 + согласованные решения пользователя.

## 0. Жёсткие ограничения (НЕ нарушать)

- **Никаких новых сущностей**: новых таблиц, новых kinds, новых API-эндпоинтов — НЕТ.
  Используем существующие: `customer_report_block`, `/api/customer-daily/blocks`,
  `/api/adaptation-report/analyze`, `/api/customer-daily/segment-analysis`,
  `/api/customer-daily/segment-comparison`.
- **Изоляция блоков по `kinds`** (CLAUDE.md): каждый запрос блоков передаёт `&kinds=...`
  с набором именно этой главы. Смешение глав = баг.
- **Snapshot-only рендер** (R2): `chapter_render.js` читает только `data_snapshot`,
  не ходит в БД/сервисы. Новых API-вызовов в рендер не добавлять.
- **TZ +5ч**: даты при редактировании блока читать из `params` снапшота, НЕ из DOM после
  roundtrip (риск сдвига через `toISOString()`).
- **PROTECTED**: `customer_daily.html` не трогать без явного запроса. Общий компонент
  проектировать подключаемым (один `<script>` + минимальная вставка HTML), не переписывая файл.
- **Весь существующий функционал сохраняется**. Все текущие связи остаются.
- Минимальные, хирургические правки. Логика анализа (`wz5RunAnalysis`, `wz5AttachBlock`,
  `wz5LoadAttachedBlocks`) НЕ меняется — переезжает в новый UI-каркас.

## 2. Целевая структура страницы (единый паттерн)

Сверху вниз, 3 зоны:

```
┌─────────────────────────────────────────────┐
│ ЗОНА 1. Выбор интервала                       │
│  • Шкала времени (TimelineBuilder)            │
│  • Пресеты быстрого выбора (3/10 сут/журнал/  │
│    вручную) — кнопки, не разрозненные радио   │
│  • Поля from/to (ручная правка)               │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│ ЗОНА 2. Виды анализа — ПЛИТКИ (catalog)       │
│  Каждая плитка: иконка/бейдж + название +     │
│  описание (что делает). Иерархия:             │
│   [ Полный анализ адаптации ]                 │
│       └ [R★ окно]  └ [ИРВ реагентов]  (вложен)│
│   [ Сегментный анализ ]                       │
│   [ Сравнение участков ]                      │
│  Клик → fullscreen-модалка анализа            │
└─────────────────────────────────────────────┘
┌─────────────────────────────────────────────┐
│ ЗОНА 3. Плитки-результаты (отчёт)             │
│  Снапшот-блоки, сгруппированы по типу,        │
│  drag-n-drop внутри группы. На плитке:        │
│  kind-бейдж, заголовок, дата, 1-2 ключевые    │
│  метрики, чекбокс in_report, кнопки           │
│  (открыть/PDF/удалить). Empty-state с подсказ.│
└─────────────────────────────────────────────┘
```

**Решение проблемы визуального шума:** Зона 2 (плитки) и Зона 3 (результаты) видны всегда
и компактны. Параметры + полный результат анализа (графики, таблицы ИРВ 30+ строк) больше
НЕ сыплются инлайн в страницу — они открываются в fullscreen-модалке по клику. Страница
перестаёт быть «простынёй».

## 3. Общий переиспользуемый компонент vs специфика главы

### 3.1 Общее (переносится на все 3 главы без изменений)

**A. Конфиг-реестр видов анализа** — `ChapterAnalysisCatalog` (чистый конфиг, без логики):
```js
{
  chapter: 'adaptation',
  blocksApiKinds: 'adaptation_period_analysis,optimal_window,reagent_irv_summary,adaptation_comparison,segment_analysis',
  analysisTypes: [
    { id, kind, label, description, icon, children:[...], runner, renderer, attacher }
  ]
}
```
- `runner` / `renderer` / `attacher` — ссылки на СУЩЕСТВУЮЩИЕ функции главы
  (для Адаптации: `wz5RunAnalysis`, `wz5RenderResult`, `wz5AttachBlock`).
- `description` — текст «что делает анализ» (см. §6).

**B. Рендер плиток каталога** — `renderAnalysisCatalog(config, containerEl, onSelect)`.
Рисует плитки + вложенные (с отступом). Вложенные плитки disabled до получения результата
родительского анализа (E5).

**C. Fullscreen-модалка** — один `<dialog>` на страницу, 3 зоны:
- шапка: kind-бейдж + название + кнопка закрытия;
- тело: блок параметров (сверху) + область результата (снизу, скролл);
- футер: «Добавить в отчёт» / «Закрыть»; в режиме редактирования плитки также «Пересчитать».

**D. Рендер плитки-результата** — `renderResultTile(block)`: бейдж, заголовок, дата,
ключевые метрики, чекбокс in_report, кнопки. Переиспользует `_kindBadge`, `PART_LABELS`.

**E. drag-n-drop внутри группы** — порт логики из `customer_daily.html` (~30 строк):
HTML5 DnD на плитках, при drop пересчёт `sort_order`, PUT для затронутых блоков.
Сортировка остаётся: `KIND_ORDER[kind]` → затем `sort_order` внутри группы.

**F. Реестр описаний** — `KIND_DESCRIPTIONS` (расширение существующего `_autoDesc` из
`customer_daily.html` до всех kinds всех глав).

### 3.2 Специфика главы «Адаптация» (уникально)

- Конкретный `analysisTypes` (5 видов, см. §4).
- Существующие функции: `wz5RunAnalysis`, `wz5RenderResult` (+ `wz5RenderCards/Charts/IRV/RStar`),
  `wz5AttachBlock`, `wz5LoadAttachedBlocks` — НЕ переписываются, вызываются из каркаса.
- TimelineBuilder уже подключён (`#wz5-timeline-container`).

### 3.3 Специфика при будущем переносе

- **Наблюдение** (шаг 3): `analysisTypes` = baseline/period/segment/segment_comparison;
  логика в `observation_ui.js`. Скрытый `wz3obs-constructor` — оставить.
- **Заказчик** (PROTECTED): `analysisTypes` = period_analysis/baseline/comparison/criteria_rose/
  segment_analysis; подключение без переписывания файла.

## 4. Каталог видов анализа главы «Адаптация»

| Плитка | kind | Уровень | runner | Endpoint |
|--------|------|---------|--------|----------|
| Полный анализ адаптации | `adaptation_period_analysis` | основной | `wz5RunAnalysis` | `/api/adaptation-report/analyze` |
| Окно R★ (оптимальный режим) | `optimal_window` | вложенный | данные из ответа `/analyze` (`r.candidates`) | — |
| Сводка ИРВ по реагентам | `reagent_irv_summary` | вложенный | данные из ответа `/analyze` (`r.irv`) | — |
| Сегментный анализ | `segment_analysis` | основной | `ObservationSegmentWidget` | `/api/customer-daily/segment-analysis` |
| Сравнение участков | `adaptation_comparison` | основной | `wz5cmp*` | `/api/customer-daily/segment-comparison` |

Вложенные R★ и ИРВ:
- Отображаются как отдельные плитки с отступом под «Полным анализом», статус «будет вычислено».
- Disabled, пока «Полный анализ» не выполнен (их данные приходят одним ответом).
- После выполнения → активны, кнопка «Добавить отдельно» создаёт самостоятельный блок
  (отдельный kind в `customer_report_block`, как сейчас).

## 5. Поток взаимодействия (по шагам)

1. **Выбор интервала** (Зона 1): шкала / пресет / ручной ввод → `from/to` в `state.steps[5].data`.
2. **Клик по плитке анализа** (Зона 2) → `dialog.showModal()`; в модалку подставляются
   параметры выбранного вида + (для повторного открытия) текущий интервал.
3. **В модалке**: регулировка параметров → «Запустить» → loading-state (спиннер +
   «вычисляю…») → результат рендерится в теле модалки (существующий `wz5RenderResult`).
4. **Футер модалки**: «Добавить в отчёт» → `wz5AttachBlock(kind, r)` → снапшот →
   POST `/api/customer-daily/blocks` (с корректным kind) → `dialog.close()`.
   «Закрыть» → закрыть без сохранения.
5. **Плитка-результат** появляется в Зоне 3 (`wz5LoadAttachedBlocks` → `renderResultTile`).
6. **Клик по плитке-результату** → модалка в режиме просмотра (снапшот + настройка частей
   вывода `in_report`/части блока + заголовок). Кнопка «Пересчитать» → режим редактирования
   параметров → повторный запуск → новый снапшот замещает старый блок (тот же block id).
7. **drag-n-drop** плиток-результатов внутри группы → `sort_order` → PUT.
8. **Превью главы** (панель 60/40, `chapter_render.js`) обновляется из снапшотов блоков.

## 6. Описания плиток (UX, на русском, осторожные формулировки)

Помещаются в плитку (2–3 строки) и в шапку модалки. Формулировки — без категоричных
causal-claims (см. memory: diagnostic interpretation style).

- **Полный анализ адаптации**: «Сравнивает фактический режим R и оптимум R★ с базой
  наблюдения (B2) и базой заказчика (B1) за выбранный интервал. Внутри вычисляет окно R★
  и сводку ИРВ.»
- **Окно R★**: «Подбирает оптимальное окно режима (давление/частота продувок) по
  кандидатам. Возможная интерпретация — требует проверки по журналу работ.»
- **ИРВ реагентов**: «Сводка по интервалам реагентного воздействия за период: вбросы,
  длительности, метрики эффективности.»
- **Сегментный анализ**: «Делит интервал на сегменты rise/plateau/decay по сглаженной ΔP
  и считает метрики по каждому.»
- **Сравнение участков**: «Сравнивает два выбранных участка по ключевым метрикам.»

## 7. Риски и инварианты (контроль при реализации)

- **E1 (kinds)**: каждый `fetch('/api/customer-daily/blocks')` передаёт `&kinds=` главы.
- **E2 (snapshot-only)**: в `renderChapter`/рендер плиток — никаких новых API-запросов;
  всё нужное лежит в `data_snapshot` на момент сохранения.
- **E3 (порядок)**: drag-n-drop меняет только `sort_order`; KIND_ORDER сохраняется →
  перестановка работает ВНУТРИ группы по типу (согласовано).
- **E4 (TZ +5ч)**: при редактировании блока даты берём из `params` снапшота, не из DOM.
- **E5 (вложенные виды)**: R★/ИРВ disabled до результата «Полного анализа»; при ошибке
  анализа — недоступны.
- **E6 (PROTECTED)**: общий компонент подключаемый, `customer_daily.html` не переписываем.
