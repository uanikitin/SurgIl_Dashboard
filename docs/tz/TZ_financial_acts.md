### 1.2. Новая таблица прайса — `contract_price`

Хранит цены с датами вступления в силу. Смена цены = новая строка с новым `effective_from`.

| Колонка | Тип | Назначение |
|---------|-----|------------|
| `id` | PK | |
| `work_type` | text/enum | `adaptation` \| `optimization` \| `foam_dosing` |
| `well_id` | FK wells, **nullable** | конкретная скважина ИЛИ NULL = цена «по умолчанию/общая» |
| `price_per_unit` | numeric(18,2) | Адаптация — **фикс. цена/скв-операцию**; Оптимизация — **цена за МЕСЯЦ** (делится на дни в расчёте); Дозирование — цена/операцию |
| `effective_from` | date | дата вступления в силу |
| `contract_ref` | text | напр. «2/24-09 от 24.09.2024» (см. ОВ-8: отдельная сущность или текст) |
| `created_at` / `created_by` | | аудит |

**Правило выбора цены** на дату `d` для (work_type, well): взять запись с
`work_type` совпадает, `effective_from <= d`, максимальный `effective_from`;
приоритет строки с конкретным `well_id` над строкой `well_id IS NULL`.
**Индекс/уникальность:** unique `(work_type, well_id, effective_from)`;
индекс `(work_type, effective_from)`.

|---------|-----|------------|
| `well_number` | text | № скважины по строке (Event.well — текст; хранить как есть) |
| `work_group` | text | `adaptation` \| `optimization` \| `foam_dosing` — группировка 3 видов |
| `unit` | text | «скв операция» \| «сут» \| «операция» |
| `price_per_unit` | numeric(18,2) | цена (из `contract_price` на дату) |

| `vat_amount` | numeric(18,2) | НДС 12 % построчно |

### 1.5. Мульти-скважинность: `Document.well_id = NULL` (АРХИТ. РЕШЕНИЕ)

`Document.well_id` — одиночный FK. Финансовый акт покрывает **все** скважины →
`well_id = NULL`, привязка скважины **построчная** (`DocumentItem.well_number`).

**Последствия (обязательно обработать):**
- `numbering.build_doc_number()` даёт `well_no="000"` при NULL → номер

- Проверить шаблоны/запросы, полагающиеся на `document.well` — защитить `{% if document.well %}`.
  (grep `document.well` по templates/ на этапе IMPLEMENT.)

### 1.6. Классификация реагентов (группы + фактическая стоимость) [ОВ-4 ✅]

Требование: справочник реагентов с группой (**ингибитор** / **пенные**) и **фактической
стоимостью** каждого реагента, редактируемый админом.

**Решение (переиспользование):** расширить существующую модель `ReagentCatalog`
(`backend/models/reagent_catalog.py` — уже есть `name`, `category`, `is_active`),
не создавать новую таблицу. Добавить колонки:

| Колонка | Тип | Назначение |
|---------|-----|------------|
| `act_group` | text/enum | `inhibitor` \| `foam` — группа для финакта |
| `unit_cost` | numeric(18,2) nullable | фактическая стоимость реагента (для учёта/изменений) |

**Правило по умолчанию:** `Super Foam` (SF) → `inhibitor`; **все остальные → `foam`**.
Админ может переназначить группу и добавлять названия реагентов в группы через
страницу настройки (§5). При подсчёте «Дозирования пенных реагентов» учитываются
только `Event`, чей `reagent` относится к группе `foam`.
> `category` не переиспользуем под группу (свободный текст, возможны коллизии) —
> вводим отдельный `act_group`.

## 2. Алгоритм авто-сборки количеств

Новый сервис `financial_act.py` (аналог `reagent_expense.py`), паттерн **rebuild**:
delete `document_items` документа → пересобрать 3 группы → insert. Переиспользовать
`_dt_range_for_month(year, month)` (импорт из `reagent_expense`).

### 2.1. Адаптация [ОВ-1 ✅]
Источник: `WellStatus (status="Адаптация")`, пересекающиеся с месяцем.
**Стоимость = фиксированная цена за скважино-операцию** (из `contract_price`), **НЕ** умножается
на количество. Колонка «К-во» — **справочная** (сколько операций было), на сумму не влияет.
Это объясняет образец: скв.85=11 и скв.74=10 → одинаковая стоимость 185 000 000,00.
`amount = price_per_unit (фикс.)`, `quantity` = число операций (информативно).
> Семантику «1 операции» для справочной колонки уточнить — **ОВ-3** (1 запись `WellStatus`
> = 1 операция / счёт событий адаптации). На сумму не влияет, поэтому не блокирует.

### 2.2. Оптимизация (клиппинг + разбивка по ценам) [ОВ-2 ✅]
Источник: `WellStatus (status="Оптимизация")`. **Формула:**
`цена_за_сутки = цена_за_месяц / дней_в_месяце`; `amount = цена_за_сутки × отработанных_суток`.
Для каждой скважины:
1. Пересечь интервал статуса `[dt_start, dt_end || now]` с `[period_start, period_end]`
   → число суток внутри месяца (образец: скв.85 «10-31.01» = 21 сут; остальные 30 сут).
   Логика близка к `WellStatus.duration_days()`, но **с клиппингом по границам месяца**
   (нужна отдельная функция клиппинга).
2. **Разбивка по интервалам цены** (`effective_from`): если в интервале обслуживания
   месячная цена менялась — разбить на подынтервалы
   `[max(status_start, price.effective_from, period_start) ..
     min(status_end, next_price.effective_from − 1д, period_end)]`,
   для каждого: `(цена_за_месяц_интервала / дней_в_месяце) × суток_подынтервала`.
   Стоимость = Σ по подынтервалам. Edge cases: смена цены день-в-день; несколько смен;
   нет цены на часть интервала.
> Проверка образца: (44 154 552 / 31 ... ) — храним **цену за месяц**, делим на дни (31 в январе).
> `1 458 064,52 = месячная / 31` ⇒ месячная ≈ 45 200 000,12; ×21 сут (скв.85) даёт
> частичный месяц. Точную месячную цену вносит админ в прайс.

### 2.3. Дозирование пенных реагентов [ОВ-4 ✅]
Источник: `Event`, фильтр `reagent IS NOT NULL AND qty > 0`, за месяц, группировка по скважине
через **строковый** join `cast(Event.well)==cast(Well.number)` (у `Event` нет `well_id`).
**Учитываются только реагенты группы `foam`** (по классификации `ReagentCatalog.act_group`, §1.6;
по умолчанию Super Foam → inhibitor, остальное → foam). Кол-во = счёт таких событий
(образец: скв.129=53, скв.131=32, скв.85=37, скв.120=9). `amount = quantity × price_per_unit`.

### 2.4. Ручная корректировка
После авто-сборки админ правит строки (кол-во/цена) в мастере до генерации
(паттерн `POST /documents/{id}/items/{id}/update` + rebuild-кнопка, как well_handover).

## 3. Расчёт и сумма прописью

- **Построчно:** `amount` по виду работ:
  - Адаптация: `amount = price_per_unit` (фикс., без × кол-ва) [ОВ-1 ✅];
  - Оптимизация: `amount = (цена_за_месяц / дней_в_месяце) × суток` (с разбивкой по ценам) [ОВ-2 ✅];
  - Дозирование: `amount = quantity × price_per_unit`.
  Далее: `vat_amount = round(amount × 0.12, 2)`; `amount_with_vat = amount + vat_amount`.
- **Итоги:** суммирование колонок построчно (НДС = Σ построчных НДС, **не** 12 % от общей).
  Правило округления — **ОВ-5** (2 знака; half-up vs banker's).
- **Сумма прописью (RU + EN):**
  - `num2words` **НЕ установлен** (проверено). Валюта «сум/тийин» нестандартна.
  - RU: требуется согласование «сум/сума/сумов», «тийин/тийина/тийинов». Вариант:
    `num2words(lang='ru')` для целой части + **кастомный** суффикс склонения валюты,
    либо полностью самописная функция. EN: число словами + «UZS and N tiyins» (проще).
  - **Выделить отдельным пунктом трудозатрат** (нетривиально) — **ОВ-9**.

## 4. Инвариант PARITY (.docx ↔ PDF)

По аналогии с правилом CLAUDE.md (HTML/PDF): **`.docx` и `PDF` обязаны содержать
идентичный контент** — те же строки/колонки/итоги/текст решения/сумма прописью.

**Архитектура:** единый источник данных
`generator._prepare_financial_act_context(document)` (аналог `_prepare_reagent_expense_context`)
→ один контекст рендерится **дважды**:
- в `.tex` (LaTeX, существующий Jinja2-env `\VAR{}`/`\BLOCK{}`) → `generate_pdf`;
- в `.docx`-шаблон (плейсхолдеры/Jinja-теги) → **новый** `generate_docx`.

Никаких повторных вычислений стоимости/НДС/прописи в двух местах — только рендер.
**Проверка перед сдачей:** собрать .docx и PDF одного месяца, сверить порисуночно/потаблично.
