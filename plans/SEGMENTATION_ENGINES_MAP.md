# Карта движков сегментного анализа (2026-06-19)

Цель: понять, есть ли «один универсальный» сегментный анализ, и оценить
целесообразность выноса в общий модуль.

## Короткий вывод
Единой функции НЕТ. Три независимых движка детекции переломов + сегментации.
`timeseries_analyzer.py` задумывался как универсальный (в шапке: «вынесен из
segment_analysis_module.py для переиспользования»), НО его импортирует только
живая страница. Два других движка его не используют.

Текстовое ОПИСАНИЕ сегментов уже унифицировано (`segment_descriptions.py`).
Не унифицирована САМА сегментация.

## Три движка

### 1. timeseries_analyzer.py — «универсальный» (живая страница /segment-analysis)
- Вход: один primary-ряд (flow_rate/p_tube/p_line/dp); secondary только для статистики.
- CP: level-shift по скользящему окну + trend-reversals; sensitivity 1–10
  (порог 25→8%, min_segment 24→6, merge 12→3); min_rsq=0.3 ОБЪЯВЛЕН, но НЕ применяется.
- Тип: initial/stable/rise/decline/sharp_* (по drift slope·days/mean, порог 5%, «резко» 30%).
- Нет: dual Q, давлений/штуцера/простоев в CP, ПАВ.
- Выход: AnalysisResult (segments[], changepoints[int], anomaly_clusters[]).

### 2. segment_analysis_module.py (+ segment_analysis_service) — Заказчик/Адаптация
- Вход: Q общий + Q рабочий (dual) + простой + P_шл + ΔP + штуцер.
- CP: level-shift (cost=|Δmean|+0.5·jump, 15%), dip-провалы, ГРАНИЦЫ кластеров
  простоев и смены штуцера = обязательные CP, частота простоев, волатильность,
  переломы по давлениям (v1.6). edge 2, merge 5, финальный фильтр >5%.
- Пороги: фикс. словарь SEGMENT_THRESHOLDS (~60 ключей) + override; шкалы 1–10 нет.
- Тип: НЕТ enum; флаги + текст (is_shutdown_cluster, is_workover, gradual_trend).
- Доменная логика: кластеры простоев, ремонт/КРС, preshutdown-вердикт, dual Q
  (common/only_total/only_working), R1-promote, δΔP=δP_уст−δP_шл (приоритет P_шл),
  ПАВ-балл (~12 признаков).
- Выход: segments[] (~30 ключей), changepoints[], descriptions[], cp_descriptions[].

### 3. observation_segment_service.py — Наблюдение
- Вход: только Q (P/ΔP/простой — только в статистику сегмента).
- CP: `_detect_changepoints` ЯВНО скопирован из движка 2 (cost=|Δmean|+0.5·jump,
  dip 25/25/7, edge 2, merge 5, фильтр ≥threshold/3). Без давлений/штуцера/волатильности.
- Чувствительность: пресеты low/medium/high/custom (порог 20/10/5%), все «[requires_calibration]».
- Тип: rising/falling/stable/insufficient_data (порог slope/день=50).
- Renderer-neutral (RFC): без текста/HTML. Простои — только маска/метаданные.
- Выход: obs_segment_v1 (versioned snapshot: block_status, quality, flags, segments[],
  changepoints[], shutdown_clusters[], diagnostics[]).

## Главное про родство
- Движки 2 и 3 — генетически родственны (ядро CP движка 3 скопировано из движка 2).
- Движок 1 — независимая переработка того же класса алгоритма (+ trend-reversals,
  своя sensitivity, но беднее по входам).
- Все три концептуально делают одно: сглаживание → level-shift → merge/filter → регрессия.

## Что мешает «склеить в одно»
1. Разный объём входов (движок 2 — мультиряд + физика причин; 1 и 3 — одноряд).
2. Несовместимые контракты выходов (mean_value vs mean_q_total vs mean_q; type vs
   флаги vs direction; 3 формата дат). obs_segment_v1 версионирован (БД, RFC).
3. Разные модели чувствительности (формулы 1–10 / словарь / пресеты).
4. Тяжёлый доменный «хвост» движка 2 (ПАВ, ремонт, dual/R1, cause) — это интерпретация,
   не детекция; в общий модуль не выносится.

## Риски объединения
- Смена порогов общего ядра меняет живые снапшоты obs_segment_v1 (БД) и customer-блоки
  → нужна совместимость снапшотов / bump schema_version + перерасчёт.
- R1-promote и ПАВ чувствительны к набору CP; тексты/cp-теги сместятся.
- Движок 2 — самый сложный и слабо покрыт тестами.

## Рекомендация (целесообразность)
Объединять стоит ТОЛЬКО ядро «детекция + сегментация» — там реальное дублирование
(движки 2/3 почти идентичны, 1 — близкий родственник). Доменную интерпретацию
Заказчика (ПАВ/cause/ремонт/dual/R1) и RFC-контракт Наблюдения оставить отдельными
слоями. Полную унификацию выходов НЕ делать (сломает БД и RFC).

Двухслойная архитектура:
- Ядро: расширить timeseries_analyzer (задействовать secondary-ряды, добавить
  «обязательные границы» — штуцер/простои, dip-детекцию, единую шкалу sensitivity,
  решить судьбу min_rsq).
- Тонкие адаптеры на потребителя: observation → obs_segment_v1 (bump schema +
  перерасчёт); customer → доменный слой поверх ядра; segment_demo уже на ядре.

Объём: ~4 файла. ОБЯЗАТЕЛЬНО golden-снапшоты «до/после» по реальным скважинам для
всех трёх потребителей + юнит-тесты ядра, ДО рефакторинга.
