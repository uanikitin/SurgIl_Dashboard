## 1. Суть проблемы (что требует заказчик)

Счёт-фактура (СФ) в PDF должна выглядеть **точно как утверждённый шаблон**
`/Users/volodymyrnikitin/Downloads/CФ 6-с апрель_3.docx`. Никаких компромиссов.

Требования к таблице работ в СФ:
1. **Таблица разбивается по страницам как обычно** (шапка + начало таблицы на
   стр. 1, продолжение на стр. 2). ЗАКАЗЧИК ЯВНО СКАЗАЛ: таблица НЕ должна
   насильно переноситься целиком на отдельную страницу.
2. **Колонка «Наименование» — объединённые ячейки** (vMerge) по группам
   (Адаптация / Оптимизация / Пенные / Ингибирующие), текст по центру.
3. **НЕТ вертикальных линий**, «протекающих» из таблицы вниз в блок подписей.
4. Выравнивание в ячейках корректное (см. §5).

## 2. Главный баг — вертикальные линии (ROOT CAUSE ПОДТВЕРЖДЁН)

При рендере .docx → PDF через **LibreOffice** (`soffice --headless
--convert-to pdf`) от нижней границы таблицы работ вниз, **через весь блок
подписей**, тянутся 2 вертикальные линии (по позициям границ колонки
«Наименование»: левый край таблицы и граница «Наименование|№ скважины»).

### Что ТОЧНО установлено экспериментами (воспроизводимо):

- **Убрать vMerge → линии исчезают полностью.**
  Тест: взял готовый `СФ-2026-05.docx`, удалил все `<w:vMerge>` из таблицы
  работ, сконвертировал — линий НЕТ. → Причина в объединении ячеек.
- **vMerge-таблица на ОДНОЙ странице (без разрыва) + текст после → линий НЕТ.**
  Синтетический тест: таблица 4×3 с vMerge в col0 + 3 абзаца после,
  всё на одной странице — линий НЕТ.
- **Вывод: линии появляются ТОЛЬКО когда vMerge-таблица РАЗРЫВАЕТСЯ между
  страницами (auto-split).** Как только LibreOffice разрывает таблицу с
  объединёнными по вертикали ячейками — последняя merged-ячейка «протекает»
  боковыми границами вниз под таблицу. Это **баг рендеринга LibreOffice**.

### Почему в исходном шаблоне заказчика бага НЕТ:

Открыл `CФ 6-с апрель_3.docx` через python-docx: таблица работ там **вручную
разбита на ДВЕ ОТДЕЛЬНЫЕ таблицы** (T1 на стр.1, T2-продолжение+«Всего» на
стр.2). Каждая таблица — замкнутая (auto-split не происходит), поэтому не течёт.
**Наша генерация делает ОДНУ таблицу через docxtpl-цикл, и она авто-разрывается.**

## 4. Возможное правильное решение (НЕ реализовано)

**Реплицировать структуру шаблона — генерировать ДВЕ замкнутые таблицы**
вместо одной авто-разрывающейся. Т.е.:
- Определить, сколько строк помещается на стр. 1 (сложно без рендера).
- Разбить строки на две группы; сгенерировать 2 таблицы, каждая со своей
  строкой-заголовком (T2 — только строка нумерации «1 2 … 10», как в шаблоне).
- Каждая таблица замкнута outer-border → LibreOffice не рвёт → не течёт.

Сложность: точка разрыва зависит от высоты рендера; нужен либо фиксированный
эмпирический лимит строк, либо два прохода. Я это НЕ осилил в разумное время.

**Альтернатива для проверки:** попробовать другой конвертер .docx→PDF
(например, docx2pdf через MS Word, или Collabora/новее LibreOffice) — возможно,
баг auto-split+vMerge там отсутствует. Стоит проверить прежде чем городить
двух-табличную генерацию.

## 5. Второй нерешённый вопрос — выравнивание / заголовок стр. 2

Заказчик: «Второй заголовок лонгтейбл должен быть только номера колонок»
(на 2-й странице при переносе повторять ТОЛЬКО строку «1 2 … 10», без строки
с названиями колонок). Я утверждал, что это «технически невозможно» (Word
требует заголовок с первой строки) и поставил полный повтор — **это тоже НЕ
то, что просил заказчик.** В шаблоне это достигнуто ручным разбиением на 2
таблицы: T2 начинается сразу со строки нумерации. Т.е. решается тем же
двух-табличным подходом из §4.
